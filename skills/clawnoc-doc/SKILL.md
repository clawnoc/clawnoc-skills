# 文档自动生成

运维操作后自动生成文档，支持 Confluence Wiki 格式和 Markdown。

## 能力

### 1. 变更记录生成
每次运维操作后自动生成变更文档。

文档模板（Confluence Wiki 格式）：
```
h1. 变更记录 — {日期} {标题}

h2. 变更概述
| 项目 | 内容 |
| 变更类型 | 配置变更 / 部署 / 故障修复 |
| 影响范围 | 服务名称、环境 |
| 执行时间 | 开始 ~ 结束 |
| 执行人 | Agent / 人工 |

h2. 变更内容
- 具体操作步骤
- 配置修改详情
- 命令执行记录

h2. 验证结果
- 健康检查结果
- 功能验证结果
- 性能指标对比

h2. 回滚方案
- 回滚步骤
- 回滚验证
```

### 2. 系统架构文档
自动采集系统信息生成架构文档。

```bash
# 采集系统信息
echo "=== OS ===" && cat /etc/os-release | head -3
echo "=== Services ===" && systemctl list-units --type=service --state=running --no-pager | grep -v systemd
echo "=== Ports ===" && ss -tlnp | grep LISTEN
echo "=== Cron ===" && crontab -l 2>/dev/null
echo "=== Disk ===" && lsblk
echo "=== Network ===" && ip addr show | grep 'inet '
```

### 3. 故障复盘报告
故障处理后自动生成复盘文档。

```
h1. 故障复盘 — {日期} {故障标题}

h2. 事件概述
| 发现时间 | {时间} |
| 恢复时间 | {时间} |
| 影响时长 | {分钟} |
| 影响范围 | {描述} |
| 严重等级 | P0/P1/P2 |

h2. 时间线
- T+0: 告警触发
- T+N: 排查过程
- T+N: 修复完成

h2. 根因分析
- 直接原因
- 根本原因
- 触发条件

h2. 改进措施
| 措施 | 负责人 | 截止日期 | 状态 |
```

### 4. 知识库更新
将排查经验沉淀到运维知识库。

```bash
# 知识库条目格式
# 标题: {问题现象}
# 标签: {服务名} {错误类型}
# 排查路径: 步骤1 → 步骤2 → 步骤3
# 解决方案: {具体操作}
# 预防措施: {监控/告警/配置}
```

## 使用方式

对 Agent 说：
- "帮我生成今天的变更记录"
- "写一份故障复盘报告"
- "更新运维知识库"
- "生成系统架构文档"
